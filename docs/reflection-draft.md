# Reflection Draft

Our project, Agentic Career Coach, was designed as a multi-agent internship support system on Google Cloud. The goal was not only to build a helpful assistant, but also to demonstrate the core architectural ideas required by the assignment: agent-to-agent delegation, Model Context Protocol integration, cloud deployment, and persistent data storage. We approached the project as a focused engineering exercise where each component needed to map clearly to the grading rubric and contribute to an end-to-end workflow.

One of the most important design choices was using a supervisor-specialist agent pattern. The Supervisor Agent is responsible for receiving user requests, interpreting user intent, and deciding when to delegate work. The Career Specialist Agent is responsible for task execution, including selecting the correct MCP tool and returning structured results. This separation made the system easier to reason about and aligned well with the assignment's expectation for agent-to-agent handoff. It also improved the clarity of our architecture diagram because each agent had a distinct role in the overall workflow.

To support tool use, we implemented an MCP server designed for deployment on Cloud Run. The server exposes two tools, `fetch_jobs` and `sync_pipeline`, through a JSON-RPC interface, with SSE transport included for agent integration. We intentionally kept the tool surface small so the project would remain stable and explainable. `fetch_jobs` handles internship discovery, while `sync_pipeline` manages create, update, get, and list operations for application tracking. This narrow scope allowed us to show meaningful MCP behavior without introducing unnecessary complexity.

For data handling, we made a practical tradeoff between realism and reliability. Internship search results are read from a mock JSON dataset instead of a live third-party jobs API. We chose this approach because it keeps the demo deterministic and avoids failures caused by rate limits or external service instability. At the same time, the project still demonstrates true persistence by writing internship pipeline records to Cloud Firestore. This means the system can save an internship, update its status, and retrieve the current pipeline in a way that reflects real backend state.

A key lesson from this project was that good system design often comes from careful scoping. It was tempting to add more agents, more tools, or a more complex user interface, but those additions would have increased the risk of failure during demonstration. Instead, we focused on making the required architecture work cleanly from end to end. Building around a clear request flow, from Supervisor to Specialist to MCP server to Firestore and back, helped us produce a system that is easier to explain, test, and evaluate.

In the future, this project could be extended in several ways. We could connect `fetch_jobs` to a live internship source, add OIDC-based authentication between Cloud Run services for extra security, and expand the pipeline into a richer career management dashboard. We could also personalize results based on student preferences and application history. Even in its current form, however, Agentic Career Coach demonstrates a strong foundation for agentic cloud applications and satisfies the assignment's core learning objectives.
